## Dependency notes

- 002 requires 001 because it reads the config at `session_start` and the
  command reads/writes it on invocation. Without 001 there is no loader to
  consume.
- 003 requires 001 because every new rendering knob reads from the same
  `MinimalToolcallConfig` object 001 defines. 003 does **not** require 002 —
  the rendering knobs work even if the user has not yet wired calm
  defaults/ownership toggles.
- 004 requires 001 and 003. The custom-tool decorator reads rendering
  knobs (preview lines, error mark) from 003, and the reload-safe disposal
  registry must clean up everything 002 and 003 register (spinners,
  grouping, overrides, command, event handlers).

## Findings considered and rejected

- **Porting pi-tool-display's MCP modes, RTK hints, hashline anchors, split
  diff, native user message box, word wrap.** Rejected — these are
  features pi-tool-display has that this package does not. Porting them
  would be scope creep unrelated to the user's "make this package
  configurable" goal. The config **architecture** is ported; the feature
  surface is only the knobs that correspond to behaviors this package
  already has.
- **A settings modal UI** (pi-tool-display has a full
  `config-modal.ts` settings overlay). Rejected for this round — the
  `/minimal-toolcall show|reset|preset` command covers the day-to-day
  toggles. A modal can be added later if the surface grows. Surfaced as
  a deferred idea in plan 002.
- **Hot-reloading config without `/reload`.** Rejected — pi-tool-display
  also requires `/reload` for tool-ownership changes. We match that.
  Surfaced as the escape hatch in plan 001.
